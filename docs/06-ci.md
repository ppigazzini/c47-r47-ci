# CI and the Test Harness

Every lane is a **bash script under `scripts/test/` that is the source of
truth**; the workflow YAML is a thin caller that installs a toolchain and calls
it. The same script runs unchanged on a maintainer's machine, so a CI failure is
reproducible locally - with one exception, the coverage lane, documented below:

```bash
bash scripts/test/run-smoke.sh
# log: ${HARNESS_WORK:-/tmp/c43-test-harness}/logs/smoke.log
```

[scripts/test/README.md](../scripts/test/README.md) owns the per-script detail:
what each lane does, the lane contract for adding one, the tooling patches, and
the environment overrides. This page is the map and the rules.

## Workflow to script

There are two families. The `test-*` workflows call a harness script; the
`c43-*` and `desktop-*` workflows drive upstream `make` directly.

| workflow | calls |
|---|---|
| `test-harness-smoke.yml` | `run-smoke.sh`, `run-leakscan.sh` |
| `test-leakscan.yml` | `run-leakscan.sh` |
| `test-testmem.yml` | `run-testmem.sh` |
| `test-coverage.yml` | `run-coverage.sh` |
| `test-valgrind.yml` | `run-valgrind.sh` |
| `test-warnings.yml` | `run-warnings.sh` |
| `test-staticanalysis.yml` | `run-staticanalysis.sh` |
| `test-fuzz.yml` | `run-fuzz.sh` |
| `test-fuzz-equation.yml` | `run-fuzz-equation.sh` |
| `test-fuzz-restore.yml` | `run-fuzz-restore.sh` |
| `c43-gcc-ci.yml` | `make test`, `make docs`, `make dist_*` (GCC) |
| `c43-clang.yml` | `make test`, `make docs`, `make dist_*` (Clang 22) |
| `c43-gcc-analysis.yml` | ASan+UBSan and Valgrind lanes, GCC |
| `c43-clang-analysis.yml` | ASan+UBSan and Valgrind lanes, Clang 22 |
| `desktop-linux-package.yml` | Linux packaging |
| `desktop-macos-package.yml` | `make dist_macos` |
| `desktop-windows-package.yml` | `make dist_windows` |

Adding a lane means writing the script first. The workflow must contain no
logic that cannot be run locally.

## The lanes

Each script syncs a fresh upstream c43 tree from GitLab on every run - nothing
is built from this repo's own sources. `scripts/test/lib/common.sh` provides the
preamble (resolve/sync upstream with submodules, xlsxio build, ccache, logging),
all environment-overridable (`UPSTREAM_COMMIT`, `HARNESS_WORK`, `LOG_DIR`, ...).

| Lane | Script | Baseline | Gate |
|---|---|---|---|
| smoke | `run-smoke.sh` | - | build surface present |
| leakscan | `run-leakscan.sh` | `leakscan-baseline.txt` (39) | **hard** |
| testmem | `run-testmem.sh` | `testmem-baseline.txt` (20) | **hard** |
| coverage | `run-coverage.sh` | `coverage-floors.txt` (5 sectors) | **hard in CI only** - see below |
| valgrind | `run-valgrind.sh` | `valgrind-baseline.txt` (8) | **hard** - `VALGRIND_GATE` defaults to **1** |
| fuzz decode | `run-fuzz.sh` | - | report-only (`FUZZ_GATE=1` to gate) |
| fuzz equation | `run-fuzz-equation.sh` | - | report-only |
| fuzz restore | `run-fuzz-restore.sh` | - | report-only |
| warnings | `run-warnings.sh` | `warnings-baseline.txt` (294) | report-only (`WARN_GATE`) |
| static analysis | `run-staticanalysis.sh` | `cppcheck-baseline.txt` (22) | report-only (`ANALYSIS_GATE`) |

Plus four upstream build/analysis lanes that use `make`, not the harness:
`c43-gcc-ci.yml`, `c43-clang.yml` (build, `make test`, `make docs`, packaging),
`c43-gcc-analysis.yml`, `c43-clang-analysis.yml` (ASan+UBSan simulator lane plus
a report-first Valgrind lane over `./c47 --writeexportall`).

### The one place the script default and CI disagree

`VALGRIND_GATE` is the **only** gate knob whose *script* default is on
(`run-valgrind.sh:32`, `"${VALGRIND_GATE:-1}"`). Every other knob -
`WARN_GATE`, `ANALYSIS_GATE`, `FUZZ_GATE`, `SECTOR_GATE`, `COVERAGE_MIN` -
defaults **off** in its script.

**The coverage lane is the exception to "the script is the contract".**
`run-coverage.sh` defaults `COVERAGE_MIN=0` and `SECTOR_GATE=0`, i.e. report
only, but `test-coverage.yml:34-35` sets `COVERAGE_MIN: "45"` and
`SECTOR_GATE: "1"` in the workflow environment. So:

```bash
bash scripts/test/run-coverage.sh                        # reports, never fails
COVERAGE_MIN=45 SECTOR_GATE=1 bash scripts/test/run-coverage.sh   # what CI runs
```

Running the bare script locally will **not** reproduce a CI coverage failure.
Either export those two variables, or read `test-coverage.yml` first. Every
other lane reproduces from the bare script.

Upstream's own `.gitlab-ci.yml` `test` stage runs only `make test` and
`make docs`. It defines no ASan, Valgrind, fuzzing, static-analysis or coverage
job - every lane above exists only here.

**Baselines are keyed on identity, not magnitude** (`leakscan LEAK item=1202`,
`keyscan CRASH seq=integrate_pgm_x20`, `eq_cov.txt|IN: FARG=0 RX=REAL:"0"`), so
counts drift without churning the gate. `UPDATE_BASELINE=1` **never overwrites**
- it writes `<baseline>.new` and returns 0. Copy it over by hand, after reading
it. The ratchet rule is in [05-debugging.md](05-debugging.md).
