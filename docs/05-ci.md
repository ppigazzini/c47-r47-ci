# CI and the Test Harness

Every lane is a **bash script under `scripts/test/` that is the source of
truth**; the workflow YAML is a thin caller that installs a toolchain and calls
it. The same script runs unchanged on a maintainer's machine, so a CI failure is
reproducible locally - with one exception, the coverage lane, documented below:

```bash
bash scripts/test/run-smoke.sh
# log: ${HARNESS_WORK:-${RUNNER_TEMP:-/tmp}/c43-test-harness}/logs/smoke.log
```

[scripts/test/README.md](../scripts/test/README.md) owns the per-script detail:
what each lane does, the lane contract for adding one, the tooling patches, and
the environment overrides. This page is the map and the rules.

## Which lane to run first

Keyed by what you just changed upstream, because that is what a reader knows
and a lane name is not. Run the first entry before the rest: it is the one most
likely to catch the class of bug that change introduces.

| you changed | run first | then |
|---|---|---|
| anything at all, as the floor | `make test` in the clone | the lane below |
| a maths, register or conversion path | `run-leakscan.sh` | `run-testmem.sh` |
| code that writes into a pool block (matrix, vector, string) | the pool canary **by hand** - [04-debugging.md](04-debugging.md) Section 5 | `run-leakscan.sh` |
| a parser, decoder, or anything reading a file or a program | `run-fuzz.sh` | `run-fuzz-equation.sh`, `run-fuzz-restore.sh` |
| the grapher, the fonts or the blitter | `make test` - `graphs_cov.txt` is the only corpus file that asserts the screen | `run-coverage.sh` |
| the keyboard, a softmenu or the display | `run-ui.sh` | a `./c47` keyboard script under `xvfb-run` |
| a lane script or a workflow in this repo | `run-docs-lint.sh` | the lane you touched, twice - see the negative-control rule in [AGENTS.md](../AGENTS.md) |
| a tracked doc | `run-docs-lint.sh` | nothing; no gate reads a sentence |
| nothing here, but a lane went red | pin `UPSTREAM_COMMIT` and re-run | bisect upstream with that lane |

The pool canary is **not a lane**. It is a patch you apply by hand, and it is
the only detector that sees an overrun inside the pool - the class ASan and
Valgrind are structurally blind to. A green CI run says nothing about it.

## The lanes

Each script syncs a fresh upstream c43 tree from GitLab on every run - nothing
is built from this repo's own sources. The one exception is the docs lane,
which reads only this repo and so needs no clone and no toolchain.
`scripts/test/lib/common.sh` provides the preamble (resolve/sync upstream with
submodules, xlsxio build, ccache, logging), all environment-overridable
(`UPSTREAM_COMMIT`, `HARNESS_WORK`, `LOG_DIR`, ...).

| Lane | Script | Baseline | Gate |
|---|---|---|---|
| smoke | `run-smoke.sh` | - | build surface present |
| docs | `run-docs-lint.sh` | - | **hard** - links, paths, pinned counts, ASCII, `__DEV/` citations |
| leakscan | `run-leakscan.sh` | `leakscan-baseline.txt` (30) | **hard** |
| testmem | `run-testmem.sh` | `testmem-baseline.txt` (11) | **hard** |

| valgrind | `run-valgrind.sh` | `valgrind-baseline.txt` (8) | **hard** - `VALGRIND_GATE` defaults to **1** |
| fuzz decode | `run-fuzz.sh` | - | report-only (`FUZZ_GATE=1` to gate) |
| fuzz equation | `run-fuzz-equation.sh` | - | report-only |
| fuzz restore | `run-fuzz-restore.sh` | - | report-only |
| ui | `run-ui.sh` | - | **hard** - each `scripts/test/ui/*.t47` must exit 0 |

Plus four upstream build/analysis lanes that use `make`, not the harness:
`c43-gcc-ci.yml`, `c43-clang.yml` (build, `make test`, `make docs`, packaging),
`c43-gcc-analysis.yml`, `c43-clang-analysis.yml` (ASan+UBSan simulator lane plus
a report-first Valgrind lane over `./c47 --writeexportall`).

### Gate state, and which of them is a decision

A lane sits in one of three states. Read this before "fixing" a report-only
lane into a gate: two of the three states are deliberate, and the third is not
documented as anything.

| state | lanes | what it means |
|---|---|---|
| **hard gate** | smoke, docs, leakscan, testmem, valgrind, ui | fails the build on a finding, from the bare script |
| **gated in CI only** | coverage | the script reports; `test-coverage.yml` supplies the gate |
| **report-only** | fuzz decode, fuzz equation, fuzz restore, warnings, static analysis | surfaces findings, never fails; a `*_GATE=1` knob turns each on |

The first two states are deliberate and their reasons are written down: the
valgrind default is on (`run-valgrind.sh:32`), and the coverage split is the
documented exception to "the script is the contract".

**For the five report-only lanes, no reason is recorded anywhere.** The scripts
call themselves "report-first" and stop there; neither they nor this page says
whether that is a permanent judgement (fuzz findings are nondeterministic, so a
gate would flap), a temporary one (the warning and cppcheck baselines describe
upstream code this repo cannot fix), or an oversight. Those are guesses, and
they are recorded here as guesses. Do not promote one to a rationale by writing
it down as fact - find out, then state it with the evidence. A report-only lane
whose reason nobody knows is a gap, not a design, and describing it as a design
is what would keep it alive.

### Running lanes locally

**One lane at a time, or give each its own `HARNESS_WORK`.** Every lane calls
`harness_sync_upstream`, which does `rm -rf "$UPSTREAM_DIR"`, and
`UPSTREAM_DIR` defaults to `$HARNESS_WORK/upstream` - one shared tree at
`${RUNNER_TEMP:-/tmp}/c43-test-harness` - `/tmp/...` locally, but
`$RUNNER_TEMP/...` on a GitHub runner, which sets it. Two lanes at once
therefore delete each other's source
mid-build; the symptom is an unrelated-looking build failure in whichever lane
was slower. A distinct `BUILD_DIR` does not help, because the tree underneath it
is what gets wiped.

```bash
HARNESS_WORK=/tmp/w-leak bash scripts/test/run-leakscan.sh &
HARNESS_WORK=/tmp/w-fuzz bash scripts/test/run-fuzz-equation.sh &
```

**Know what a lane costs before you wait on it.** The Valgrind lane runs 2-3
hours (measured: 132, 135 and 199 minutes on three green runs) - three of them
sitting `in_progress` for over an hour is normal, not a hang. Check
`gh run list --workflow="Valgrind Memcheck"` for the current figure rather than
trusting this sentence. The docs lane takes seconds; the leak, testmem and fuzz
lanes take minutes.

**`UPSTREAM_COMMIT` makes any lane a bisect probe.** It pins the lane to a full
40-character upstream SHA, fetched `--depth 1`, so a lane can be the test in
`git bisect run` over upstream history - that is how the two new leak-gate
crashes were attributed to the commit that exposed them.

### The one place the script default and CI disagree

`VALGRIND_GATE` is the **only** gate knob whose *script* default is on
(`run-valgrind.sh:32`, `"${VALGRIND_GATE:-1}"`). Every other knob -
`WARN_GATE`, `ANALYSIS_GATE`, `FUZZ_GATE`, `SECTOR_GATE`, `COVERAGE_MIN` -
defaults **off** in its script.

**The coverage lane is the exception to "the script is the contract".**
`run-coverage.sh` defaults `COVERAGE_MIN=0` and `SECTOR_GATE=0`, i.e. report
only, but `test-coverage.yml:34-35` sets `COVERAGE_MIN: "45"` and
`SECTOR_GATE: "1"` in the workflow environment. So:

```bash
bash scripts/test/run-coverage.sh                        # reports, never fails
COVERAGE_MIN=45 SECTOR_GATE=1 bash scripts/test/run-coverage.sh   # what CI runs
```

Running the bare script locally will **not** reproduce a CI coverage failure.
Either export those two variables, or read `test-coverage.yml` first. Every
other lane reproduces from the bare script.

Upstream's own `.gitlab-ci.yml` `test` stage runs only `make test` and
`make docs`. It defines no ASan, Valgrind, fuzzing, static-analysis or coverage
job - every lane above exists only here.

**Baselines are keyed on identity, not magnitude** (`leakscan LEAK item=1202`,
`keyscan CRASH seq=integrate_pgm_x20`, `eq_cov.txt|IN: FARG=0 RX=REAL:"0"`), so
counts drift without churning the gate. `UPDATE_BASELINE=1` **never overwrites**
- it writes `<baseline>.new` and returns 0. Copy it over by hand, after reading
it. The ratchet rule is in [04-debugging.md](04-debugging.md).
